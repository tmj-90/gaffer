# Security model

Gaffer runs **shell-capable autonomous agents** on your machine, against your repos, with your API keys. That is powerful and inherently risky. This document is honest about both what protects you and where the limits are — read it before pointing Gaffer at anything you don't fully trust.

## Threat model in one line

You are trusting an LLM agent with a shell. The agent's *inputs* (ticket text, repo contents, ingested issues) may be attacker-influenced; the agent itself may be prompt-injected. Gaffer's job is to **contain what the agent can do** and **keep a human in the loop on anything that ships**.

## What protects you (defense in depth)

- **Deterministic PreToolUse safety hook** (`runner/safety-hook.mjs`): scopes all writes to the active worktree (`GAFFER_WRITE_ROOTS`), denies reads of secret files by *path* (not a reader-binary denylist), denies the control-plane CLI and raw DB access to the agent — including a **self-spawned control-plane MCP** (`node …/dispatch/dist/mcp`), `node -e` reaching the dist or the DB, and reading another process's environment (`ps`, `/proc/*/environ`) to lift a token — blocks force-push / history-execution git config / scheduled-execution primitives, and resolves wrapped/assignment-prefixed commands so `env tee`, `VAR=1 mv`, `sh -c '…'` can't smuggle a verb past it. It **fails closed**: when it can't prove a target is in-scope, it blocks.
- **Throwaway git worktrees**: each ticket runs in a disposable worktree; the real checkout is never touched. Teardown runs on the explicit success/error paths **and** from an `EXIT`/`INT`/`TERM` trap, so a crash or signal mid-delivery cannot orphan a worktree or a half-finished `gaffer/ticket-*` branch — the trap is guarded so a legitimately-delivered branch is preserved for review/merge.
- **Server-side human review gate**: the merge gate is enforced in the control plane, not by convention. **By default** an `agent`-type actor cannot approve a review or grant itself repo access; in factory context the agent-facing MCP tools additionally refuse to **submit or claim** a ticket even with an explicit token (submission is runner-owned), so a disobedient agent cannot self-submit a no-op delivery past the DoD gates. The "done" gate verifies the **real `git diff`** — not an agent-authored evidence string. An agent-supplied `pr_url` is treated as an evidence/navigation link **only**; it never substitutes for the server-recomputed `git diff`, so a bogus URL cannot satisfy the gate. The opt-in autonomy flags (`DISPATCH_ALLOW_AGENT_APPROVE`, `MERGE_ON_AGENT_REVIEW`, `MEMORY_AUTO_APPROVE`) deliberately relax this — they are **off by default**, and enabling one removes the corresponding human gate. Treat them as a conscious trust decision, not a convenience toggle.
- **Enforced Definition of Done (I3)**: before a delivery may rest in the human review lane, the **runner** (never the agent, which would be trusting the fox to count the hens) runs the enabled DoD gates — **tests / typecheck / lint** — deterministically in the delivery worktree, each bounded by `gaffer_timeout`. A failing gate **auto-rejects the delivery back to rework** with the gate name + a captured output tail recorded as evidence; a human never spends time on a delivery that doesn't pass its own gates. This turns review from "read all the code and hope" into "audit a pre-verified delivery." It is configurable per repo and factory-wide (`definition_of_done` in `crew.yaml`); a gate with **no configured command is skipped, not failed** (and logged); `GAFFER_DOD=0` disables enforcement (the pre-I3 behaviour). **Shipped gates:** tests · typecheck · lint. **Deferred (documented follow-ups, not yet gates):** coverage-did-not-decrease (needs a stored baseline), SAST/SCA (needs I2), CI-green (H3), docs-updated. The gate is a *quality* barrier, not a security boundary — the human review gate above remains the thing that actually keeps a human in the loop on anything that ships.
- **Prompt-injection handling**: the untrusted free-text ticket fields the agent reads through `get_ticket` — **title, description, acceptance-criterion text, and review feedback** — are wrapped server-side in `<untrusted-*>` quarantine envelopes carrying a standing "this is data, not instructions" notice; the runner prompts and skills carry the same framing, and event/comment payloads are redacted out of the agent's context entirely rather than enveloped. The live agent's environment is built from an **allowlist** (`env -i` + only what `claude` and its MCP tools need): every ambient credential is stripped — `GITHUB_TOKEN`, `AWS_*` access keys / secret / session token, `DISPATCH_API_TOKEN`, and any `*_TOKEN` / `*_SECRET` / `*_KEY` / `*_PASSWORD` (except `ANTHROPIC_API_KEY`, which `claude` needs for auth). This applies to **both** the shell `claude -p` launches (delivery, bootstrap, agent-review, clarify) and the `.mjs` runners (decompose, product-owner, onboard).
- **Resource caps**: per-tick wall-clock timeouts, agent turn caps, and dependency installs forced to `--ignore-scripts` (no `postinstall` code execution).
- **Optional OS sandbox** (`runner/lib/sandbox.sh`): a kernel-level **write** boundary via a provider seam — a *write sandbox*, not a full jail: it scopes where the agent can write, but does **not** yet isolate reads or network egress (see the residual limits below). `GAFFER_STRICT_REQUIRE=1` makes it **fail closed** — when no OS sandbox provider is available (any non-macOS host, or a missing `sandbox-exec`), the runner refuses to launch the agent rather than silently degrading to "no OS sandbox". **Enabling any autonomy flag auto-defaults `GAFFER_STRICT_REQUIRE=1`** (see below), so unattended runs fail closed without an OS sandbox unless you explicitly opt out.
- **Planning agents run read-only.** The propose/analyse agents (`decompose`, `spec-author`, `onboard-analyze`) run **hook-less in the factory's OWN source dir** (`RUNNER_DIR`) — a prompt injection in an untrusted brief or onboarded-repo file could otherwise write to factory source. They are launched with the **complete write/exec tool set denied** — `--disallowedTools Write Edit MultiEdit NotebookEdit Bash` — leaving only Read/Grep/Glob plus the MCP tools they connect via `--mcp-config`. **Design decision (kept):** a *denylist* of write tools, **not** an `--allowedTools Read Grep Glob` allowlist — because these agents connect MCP, and an allowlist would sever those tools (or require enumerating every MCP tool name). The residual: a *new* write tool added to Claude Code must be added to the denylist. That is covered by a periodic audit of Claude Code's write/exec tool set (and the agent-spawn tests pin the current denied list); if Claude Code ever exposes a stable `--allowedTools` MCP-wildcard, revisit the allowlist as the more robust form.

## Honest residual limits (the ceiling)

- **The safety hook is a regex/string analyzer on bash.** It is hardened and fails closed, but a string bouncer fundamentally cannot win against a determined agent that constructs paths dynamically, decodes arguments at runtime, or writes via a child process's syscalls. The deterministic boundary shrinks the residual to dynamic *reads* and fully-dynamic non-secret *paths* — it does not eliminate it.
- **The only working OS sandbox provider today is `sandbox-exec`** — macOS-only, Apple-deprecated, and write-only (no per-subprocess network isolation). Container/VM providers exist as a seam but are stubs. **On any non-macOS host, the deterministic boundary is the regex hook alone.** A direct corollary: the `strict` autonomy mode adds only this OS sandbox on top of `autonomous`, so **on a non-macOS host `strict` behaves exactly like `autonomous`** — there is no extra containment. Use `GAFFER_STRICT_REQUIRE=1` to fail closed instead of silently degrading.
- **Worker provider containment — the only supported worker is Claude Code.** The runner reaches its agent through a provider seam (`GAFFER_WORKER_PROVIDER`, default `claude-code`) that mirrors the sandbox seam. This is deliberately a **seam, not a second worker**: the one real provider is `claude-code`, and any other value (`codex`, `local`, …) is an honest stub that **fails closed** — it refuses to run, returns non-zero, and never spawns an agent (`worker provider <x> not yet supported; safety-hook containment unavailable`). The reason it fails closed rather than degrading (the way a missing OS sandbox degrades) is the load-bearing tension: **the PreToolUse safety hook is Claude-Code-native** — it runs *in-process* inside `claude -p`. A non-Claude worker has **no equivalent in-process containment**, so it cannot run unattended at the same posture until a real OS-sandbox provider (container/VM) exists to supply the boundary the hook otherwise provides. Enabling a non-Claude provider today therefore does nothing except surface that gap loudly. A second consequence lives one layer down: the worker's result parser (`lib/worker.mjs` `parseResult`) is **Claude-envelope-specific** — it decodes the `claude -p … --output-format json` shape. A non-Claude provider needs its **own** parser feeding the same `{resultText, usage, capHit, stopReason}` contract before its output could be trusted by the ledger and the cap/spend guards.
- **No network egress isolation** when the agent process is wrapped as a whole. An agent that can run code can reach the network.
- **Notification sinks are an outbound channel.** The webhook (`GAFFER_NOTIFY_WEBHOOK_URL`) and Slack (`GAFFER_NOTIFY_SLACK_URL`) sinks POST human-gate events to a URL you configure. Outbound bodies are **redacted by default** — a minimal body (kind · ticket number · status · dashboard URL only), dropping the ticket `title`/`detail` free text a prompt-injected agent can influence, so an untrusted endpoint is not an exfiltration path. Set `GAFFER_NOTIFY_FULL_PAYLOAD=1` to include the full free-text body — only when the endpoint is inside your trust boundary. (The old opt-in-to-redaction `GAFFER_NOTIFY_REDACT` is deprecated; `GAFFER_NOTIFY_REDACT=0` is still honoured as a full-payload request.) Treat the notify URL like any other outbound integration — point it only at endpoints you trust.
- **The opt-in autonomy flags remove protections by design.** `DISPATCH_ALLOW_AGENT_APPROVE` / `MERGE_ON_AGENT_REVIEW` let the agent ship without a human; `MEMORY_AUTO_APPROVE` lets agent-authored memory go live immediately, which re-opens cross-ticket memory poisoning. Only enable these with **trusted input**.

## Running it safely

- Keep the human review gate **on** for anything that matters.
- Do **not** point Gaffer at untrusted issues/repos (or enable the autonomy flags) without a real sandbox provider and a machine whose blast radius you accept.
- **For unattended autonomy specifically** (`GAFFER_MODE=autonomous`/`strict`, or the `DISPATCH_ALLOW_AGENT_APPROVE` / `MERGE_ON_AGENT_REVIEW` / `AUTO_MERGE` / `MEMORY_AUTO_APPROVE` flags), the **OS sandbox is the containment boundary** for a determined or prompt-injected agent — in the default supervised mode, the human review gate is that boundary instead. The runner **enforces this in code**: enabling any of those autonomy flags **auto-defaults `GAFFER_STRICT_REQUIRE=1`**, so the runner fails closed (refuses to launch) when no OS sandbox provider is available — unless you *explicitly* set `GAFFER_STRICT_REQUIRE=0` to assert you've supplied containment out-of-band (a VM/container around the whole runner), which is logged loudly. So: run unattended only where an OS sandbox is available (macOS `sandbox-exec` today) or with explicit out-of-band containment — otherwise keep a human on the gate. (Graduated mode ships only what each repo has *earned* via per-repo/risk policy; the same containment expectation applies at its per-ticket ship gate.)
- Treat the host like one running an autonomous process with shell access: isolated user, no ambient cloud credentials, scoped repo access. The agent's environment is allowlist-scrubbed before each launch, but that is defence in depth — keeping host credentials out of the runner's own environment remains the right posture.
- Prefer running against repos you own, on a machine you control. That is the supported, local-first posture.

## Reporting

This is run-at-your-own-risk software (see [`LICENSE`](LICENSE) — Apache-2.0, no warranty). If you find a vulnerability, please report it privately through **GitHub's private vulnerability reporting** — open the repository's **Security** tab and choose **"Report a vulnerability"** (Security → Advisories → Report a vulnerability) — rather than filing a public issue. Include a reproduction and the host/OS context. Private reporting keeps the details embargoed until a fix is available.
